# Техническое задание: Разработка инструмента сверки данных (Data Reconciliation Tool)

**Версия:** 1.0 (Финальная)  
**Дата:** 2024  
**Роль:** Senior Python Developer / Data Engineer  

---

## 📌 Задача
Создать скрипт в Jupyter Notebook для выверки данных между **Oracle** (Источник) и **Postgres** (Приемник) в рамках реализации КХД (Форма 110).

**Объем данных:** 3–5 млн строк  
**Особенности:** PK и индексы отсутствуют

---

## 1. Входные параметры (Config)

Скрипт должен принимать объект конфигурации со следующей логикой:

### 1.1. Connections
- Данные доступа к Oracle и Postgres

### 1.2. Targets
- Возможность указать **разные схемы и имена таблиц** для каждой БД  
  *Пример:* `ORA_SCHEMA.TABLE_V1` и `PG_SCHEMA.TABLE_V2`

### 1.3. Composite Keys
- Список полей бизнес-ключа для группировки  
  *Пример:* `['BANK_CODE', 'REPORT_DATE']`

### 1.4. Exclusions (Исключения)
- Список полей, которые **не должны участвовать** в сверке (технические ID, даты загрузки и т.д.)
- **Важно:** Если список не указан (`None` или пустой), в сверку должны попадать **все доступные поля** таблицы

### 1.5. Фильтрация и Sample
- **sample_size** (`int`):  
  - Если указано число `N`, выбрать `N` случайных уникальных комбинаций бизнес-ключа для проверки
  - Если `None` — проверять всё
- **specific_keys** (`list/dict`):  
  - Возможность передать конкретные значения ключей (напр. список кодов банков) для точечного анализа

---

## 2. Автоматическое определение типов (Intelligent Mapping)

Скрипт должен **самостоятельно анализировать структуру таблиц**:

### 2.1. Числовые поля
- Все типы: `numeric`, `decimal`, `float`, `number` (кроме тех, что в Exclusions)
- Участвуют в расчете **SUM()**

### 2.2. Строковые/Лексические поля
- Все текстовые типы: `varchar`, `text`, `char`
- Участвуют в сверке «лексики»
- Автоматически применяется: `TRIM(UPPER(CAST(col AS TEXT)))`

### 2.3. Даты
- Все поля типа `date`/`timestamp`
- Приводятся к формату **YYYY-MM-DD**

## 3. Двухэтапный алгоритм сверки

### Этап 1: Агрегированная проверка (Макро-уровень)
1. На стороне БД выполняется SQL запрос с **группировкой по Composite Keys**
2. Считается:
   - `COUNT(*)`
   - `SUM()` по всем обнаруженным числовым полям
3. Результаты из Oracle и Postgres объединяются в один DataFrame
4. Находятся расхождения (**Delta**)

### Этап 2: Детальная сверка (Микро-уровень)
1. Для ключей с расхождениями (или для выбранного `sample_size`) выкачиваются **полные данные**
2. Поскольку PK нет, внутри группы данные **принудительно сортируются** по всем полям (`sort_values`) для обеспечения идентичного порядка
3. Используется `pandas.compare()` для вывода конкретных различий в текстовых и числовых полях

---

## 4. Технические требования и нормализация

### 4.1. Обработка NULL
- Использовать `COALESCE` (или `NVL` в Oracle)
- Пустые значения должны считаться **идентичными** и не выдавать ошибку при сравнении

### 4.2. Типы данных
- Реализовать **явный CAST** в SQL запросах
- Избежать конфликтов при сравнении разных диалектов (Oracle vs Postgres)

### 4.3. Округление
- Числа округлять до **2 знаков**: `ROUND(col, 2)`

## 5. Визуализация (Seaborn)

### 5.1. Heatmap
Построить тепловую карту, где:
- **Ось X** — один из компонентов бизнес-ключа (напр. отчетная дата)
- **Ось Y** — второй компонент (напр. код банка)
- **Цвет** — статус сверки:
  - 🟢 Зеленый — OK (совпадение)
  - 🔴 Красный — расхождение (MISMATCH)
  - ⚪ Серый — данные отсутствуют в одной из баз (ONLY_IN_SOURCE / ONLY_IN_TARGET)

### 5.2. Summary Table
Вывести сводную таблицу с количеством найденных ошибок по типам:
- MISMATCH
- ONLY_IN_ORACLE
- ONLY_IN_POSTGRES
